Summary

Stand up the FrigoLoco PostgreSQL database (the foundation of the cloud ERP that replaces the Excel forecasting/return workbooks) and pull all available historical data into it: full history from the Intelligent Fridges ("Husky") API — purchases, restocks, product reviews, master data — via a resumable, chunked backfill that archives every raw API payload to blob storage, plus a one-off importer that migrates the historical facts locked inside the two Excel workbooks (≈20,700 dispatch lines, 513 purchase orders, 50+ weeks of P&L data). Incremental cron jobs then keep the database current, including periodic live-stock snapshots that cannot be backfilled later.

Why

FrigoLoco runs 40+ RFID smart fridges at corporate sites in Belgium, targeting 200. The entire operation currently lives in two Excel workbooks (Excel Files/Forecasting Tool/Smart Fridge Forecasting Tool V5.xlsx, Excel Files/Forecasting Tool/Weekly & Monthly Return V2.xlsx) automated by ~25 Office Scripts. The dev presentation (FrigoLoco_Dev_Presentation_V5_Final.pptx) commits to replacing this with a cloud ERP — React + FastAPI + PostgreSQL on Railway with Azure Blob storage.

Everything downstream (forecasting, menu, dispatch, finance) depends on one thing existing first: a database with the full history in it. Concretely:

  • Excel is at its limits. Accurate forecasting needs 20–30M RFID event rows; Excel cannot hold them. Today only 3 weeks of history feed the forecast and only pre-aggregated slices survive in the workbooks.
  • The deck's forecast algorithm needs deep history: 6–12 months of per-fridge/per-category/per-weekday sales, and 18 months of weather–sales correlation. That history exists only inside the Husky API — it must be pulled once, completely, and kept.
  • Live stock is unrecoverable. GET /stock/current is point-in-time only; the API keeps no stock history. Every day we don't snapshot it is a day of residual-stock training data lost forever.
  • The Excel workbooks hold facts the API doesn't: dispatch history (what we planned/delivered, vs. the API's what was scanned), purchase orders, and 50+ weeks of manually-entered financials (catering, TGTG, logistics costs). Retiring Excel without importing these loses the reporting baseline.
  • Cost of inaction: every new feature spec would re-solve data access; credentials stay hardcoded in 10 scripts; the 200-fridge growth target stays blocked on a spreadsheet.

Background & Context

Current system (verified by direct code/file inspection)

  • Vendor naming: the deck says "Husky"; the code calls https://api.intelligentfridges.com/api/v1/frigoloco/…. Same platform — Husky Intelligent Fridges (intelligentfridges.com). Not huskyintelligence.com (unrelated field-service SaaS).
  • Official OpenAPI 3.0.1 spec obtained from the live server (/api/openapi.json, auth-gated Swagger UI) and preserved at reference-docs/intelligentfridges_openapi_v1.json — "Intelligent Fridges API v1", 43 paths, 26 GET.
  • Auth: HTTP Basic only. Credentials are currently hardcoded in plaintext in 10 Office Scripts (mathieu@frigoloco.be + password) — must move to env config and should be rotated.
  • No pagination anywhere in the API. Event endpoints are windowed by from/to date-times (default yesterday→today); backfill must chunk by date range. No documented rate limits.
  • Money is int64 minor units (cents) in the API schema. The Office Scripts divide by 100. The database stores integer cents; formatting happens at the UI layer only.
  • Office Scripts flow: scripts pull from 6 API endpoints, write into workbook tables, and push files/emails through 4 Power Automate flows. Central fact tables: GlobalDispatchHistoryTable (~20,688 rows), OrdersSummaryTable (513 POs) + OrdersLineItemsTable (~2,584 lines), WeeklySummaryDataTable (50+ weeks, 30 columns).

Proposed Solution

Scaffold the monorepo (backend only for now; frontend/ is a placeholder for the next spec), define the full Phase-1 PostgreSQL schema with Alembic, and build four data movers on a shared ELT pattern: every API response is archived raw to blob storage first, then transformed and idempotently upserted into Postgres. A sync_run bookkeeping table makes the backfill resumable and auditable.

Blob layout (gzipped JSON, one object per fetched chunk — immutable, enables re-transforming without re-fetching):

raw/husky/{endpoint}/{YYYY}/{from}--{to}.json.gz     e.g. raw/husky/purchases/2024/2024-03-04--2024-03-10.json.gz
raw/husky/stock_current/{YYYY}/{taken_at_iso}.json.gz
raw/excel/{workbook_name}/{import_run_iso}/...       (copies of the workbooks at import time)

Backfill algorithm (purchases / restock / productreview)

  1. Master data first: sync producttype, fridge, facility, fridgegroup, fridgeproductprice so every event row can FK-link.
  2. Discover the data horizon: probe year-starts backwards (2026 → 2025 → …) with 1-week sample windows until two consecutive years return empty; the first non-empty year's start becomes BACKFILL_FROM (overridable via env, see Q1).
  3. Chunked walk: iterate 7-day windows from BACKFILL_FROM to now. Per window and endpoint: skip if a sync_run with status IN ('success','empty') already covers it (resume-for-free); fetch with retry/backoff (5 attempts, exponential, honouring 429/5xx); gzip raw payload to blob; transform; bulk upsert (INSERT … ON CONFLICT DO UPDATE on the natural keys above); write the sync_run outcome.
  4. Throttle: ≤1 request/second (no documented rate limits — be polite; ~5 years × 52 weeks × 3 endpoints ≈ 800 requests ≈ 15 min anyway).
  5. Restock is pulled with no action/status filters — ADDED, REMOVED and UNCHANGED all land in restock_tag. (The Office Scripts only ever pulled ADDED; REMOVED is required for the deck's withdrawal-list and residual-stock features.)
  6. Purchases payment-status scope is an open decision (Q9): pull unfiltered (all 15 payment statuses stored, filtered at query time) vs. only settled sales as the Office Scripts do.

Excel history importer (one-off)

python -m app.migration.import_excel reads both workbooks with openpyxl — defaulting to Excel Files/Forecasting Tool/Smart Fridge Forecasting Tool V5.xlsx and Excel Files/Forecasting Tool/Weekly & Monthly Return V2.xlsx at the repo root, overridable via --forecasting-workbook / --returns-workbook — and loads: suppliers (+emails split), category list, fridge fees/fractions/service additionals, snack&drink targets, dispatch batches+lines, purchase orders+lines, weekly summaries. Cleaning rules: strip() all status/name strings; map fridge display names to fridge.friendly_name via an explicit name_mappings.py dict (drifted names fail loudly and are added to the map by hand); product codes forced to text; euros → integer cents via round(Decimal(value) * 100); unknown product codes create is_active=false stub products so no history line is dropped; product order is preserved — product.display_order is set from the source left-to-right column order of the Menu/Dispatch sheets and never re-sorted alphabetically (deck slide 23: the "import loses product order" Excel bug must not be replicated).

Trade-off Analysis

Decision 1 — Raw-first ELT vs. direct transform

A — Archive raw to blob, then transform (ELT)
Chosen

Every API byte is kept immutably; transforms can be re-run forever without re-fetching.

  • Schema mistakes are recoverable (re-transform from blob)
  • Vendor API drift is diffable against archived payloads
  • Matches the deck's "blob for historic log data" decision
  • Extra storage moving part (mitigated: Azurite locally)
B — Transform in memory, store only Postgres rows
Alternative

Simpler pipeline, one storage system.

  • Fewer components
  • Any mapping bug forces a full re-crawl of the vendor API
  • Loses fields we chose not to model yet (nutrition, allergens, translations)

Decision: A. The API has no pagination and unknown rate limits — re-fetching 5 years of history to fix a transform bug is exactly the failure mode to design out. Risk accepted: blob credentials/config are one more manual step (see Manual Steps). Reconsider only if blob provisioning blocks the timeline; the storage layer is a thin interface either way.

Decision 2 — Backfill chunk window

Attribute7-day windows1-day windowsMonthly windows
Requests for ~5y × 3 endpoints~800~5,500~180
Payload size per responseModerate (safe)TinyLarge — risky with no pagination
Resume granularityGoodBestCoarse
Matches existing script precedentYes — scripts pull week-by-weekNoMonthly used for rating only

Decision: 7-day windows, with automatic halving on oversized-response failure. The Office Scripts already pull weeks and months successfully, so weekly payloads are proven safe.

Decision 3 — Cron execution model

A — Railway cron jobs (one-shot containers)
Chosen (pending Q4)

Each job is python -m app.jobs.X on a Railway cron schedule.

  • No always-on scheduler process to babysit
  • Per-job logs/retries visible in Railway UI
  • Jobs are plain CLIs — trivially runnable locally
  • Minimum practical cadence coarser than an in-process loop
  • Schedule lives in Railway config, not code
B — APScheduler worker container
Alternative

A long-running container schedules everything in code.

  • Schedules versioned in the repo
  • Sub-minute cadence possible
  • One crashed process silently stops ALL jobs
  • Always-on container cost

Decision: A, provisionally — every job is written as a plain CLI either way, so switching later costs one config change. If the stock-snapshot cadence chosen in Q3 is finer than Railway cron comfortably supports, revisit for that one job only.

Decision 4 — Money representation

Integer cents (bigint) everywhere, matching the API's verified int64 minor-unit schema; numeric only for rates/fractions. Rejected: numeric(8,2) euros (invites float-ish drift and double conversion — Excel's euros are converted once at import: round(Decimal(v) * 100)), and floats (never for money). This is the deck-independent choice with the fewest conversion boundaries.

Summary

Port FrigoLoco's smart-fridge operation — today spread across two Excel workbooks, three emailed template files, 26 Office Scripts, and four Power Automate flows — to a cloud ERP: React (TypeScript) frontend, FastAPI (Python) backend, PostgreSQL database, hosted on Railway with Azure Blob for generated documents. All business logic (forecasting, product scoring, menu allocation, purchase orders, stock, dispatch, RFID reconciliation, weekly/monthly P&L) moves into versioned backend services fed by scheduled syncs from the Husky RFID API.

ScopeGreenfield ERP · 8 modules · 42 fridges → 200 · 540 products
Replaces2 workbooks · 3 templates · 26 scripts · 4 PA flows
Effort5 phases · ~14–18 weeks
RiskMedium — parallel-run mitigates
DependenciesHusky RFID API · Railway · Azure Blob · email provider

Why

FrigoLoco stocks smart fridges at 40+ client sites in Belgium. Every operational decision — what to cook-order, what to load into which fridge, whether the driver actually loaded it, and whether the week made money — currently lives in Excel. The system works, but it is at its ceiling:

Pain pointEvidence in the current filesCost of inaction
Excel row limitsForecasting needs 20–30M rows of RFID event data; Global Dispatch History is already 20,692 rows and duplicated into a full backup sheet on every saveForecast accuracy capped at 3-week windows; scaling to 200 fridges is impossible
Manual data pipelinesWeekly financial reporting requires copy-pasting between the Forecasting workbook and the Return workbook, glued by Power Automate flows with hardcoded signed URLsHours of weekly manual work; silent breakage when a flow's SAS token expires
Data-integrity holesNegative stock is possible; past-date dispatches save silently; order cancellation does not reverse stock; no user/timestamp on most savesStock counts drift; no audit trail when reconciliation disputes arise
Secrets in plaintextHusky API credentials and Power Automate SAS tokens are hardcoded in every one of the 26 Office ScriptsAnyone with workbook access owns the merchant API account
Fragile positional logicScripts address data by hardcoded row/column offsets (e.g. "client rows start at row 18, score row = code row + 5"); inserting a row breaks the systemEvery layout tweak is a production incident
No concurrency / rolesOne shared workbook; warehouse, ops, and finance all edit the same grid with no permissionsCannot onboard warehouse staff or drivers as the fleet grows 5×

The June 2026 dev briefing (FrigoLoco_Dev_Presentation_V5_Final.pptx) has already committed to the target stack and module set; this spec turns that briefing plus the reverse-engineered Excel business logic into an implementable plan.

Proposed Solution

Build a single-tenant cloud ERP with the stack already selected in the dev briefing. The port is logic-first: every Office Script becomes a tested Python service; every Excel table becomes a normalized PostgreSQL table; every Power Automate flow becomes a backend job (document generation + email); every template file becomes a server-rendered PDF/XLSX. Excel stays live per module until that module is validated (parallel run).

Data model

PostgreSQL, normalized around five aggregates: catalogue (suppliers, categories, products), clients/fridges, supply (POs, stock ledger), dispatch (menus, forecasts, dispatches), and events (raw RFID sales/restock/review events synced from Husky — the 20–30M-row store Excel could never hold). Monetary columns are NUMERIC(10,2) in euros; all Husky cent/comma-string values are normalized at ingestion. All timestamps are stored as UTC timestamptz; day, week, and month bucketing (R1, R11) is computed in the operational timezone Europe/Brussels.

Dispatch lifecycle (the operational core)

stateDiagram-v2
    [*] --> Draft : apply forecast or edit manually
    Draft --> Saved : save dispatch (audit stamped)
    Saved --> Draft : reopen for edit
    Saved --> Dispatched : confirm dispatch (stock deducted, sheets emailed)
    Dispatched --> Reconciled : restock verification vs RFID ADDED events
    Reconciled --> [*]

Confirming a dispatch is one transaction: flip status, insert negative stock_movements per line, snapshot prices, generate per-fridge delivery documents (category print order per R8, withdrawal list for short-DLC products), upload to Azure Blob, and email. Saving for a past date requires an explicit confirmation flag (slide 23). Per-fridge sheets include the In-box / Missing columns the drivers use today.

Bulk dispatch-line updates accept a list of {fridge_id, product_id, qty, source} and validate against menu membership, per-fridge caps, and available stock in one round trip (red-cell feedback in the matrix UI).

Husky sync strategy

A scheduler (APScheduler in-process) replaces the on-demand fetches the scripts do today: hourly incremental pulls of purchases and restock events (idempotent upsert on husky_ref), nightly product catalogue + review sync and score recomputation, every 15 min live-stock snapshot for targets/alerts. A one-time backfill imports 12+ months of history so scoring and forecasting have their full windows from day one. All Husky credentials live in environment variables — the plaintext credentials currently in the scripts must be rotated at cutover. Failed sync runs retry with exponential backoff (3 attempts); a feed that still fails raises a sync_failure alert so a broken pipeline is visible in the alert inbox, never silent. Live stock (/stock/current) is deliberately not persisted: the 15-minute job fetches it and evaluates targets/alerts in memory, and menu/dispatch flows call it on demand (see Decision 1).

Trade-off Analysis

Decision 1 — Where does RFID event truth live?

Ingest & persist raw events in PostgreSQL
Chosen

Scheduled jobs pull Husky events into partitioned sales_events / restock_events tables; all engines query locally.

  • Solves the stated 20–30M-row pain directly; enables 6–12-month forecast windows
  • Reports and scoring are fast, reproducible, and work when Husky is down
  • One normalization point for the cents/comma-string/status quirks
  • Sync pipeline to build and monitor; storage grows ~linearly with sales
Query Husky live per request
Alternative

Replicate the scripts' pattern: fetch date-windowed data on demand.

  • No sync jobs; always-current data
  • Reproduces today's bottleneck (Update Rating already needs 12 chunked monthly pulls per run)
  • Every report is as slow and fragile as the external API
  • No history beyond what Husky retains

Decision: persist raw events. This is the core reason the platform is being built; live queries would keep the ceiling. Risk: divergence from Husky — mitigated by idempotent upserts keyed on Husky identifiers and a daily count-reconciliation job. Live-stock (/stock/current) remains a pass-through call since it is a point-in-time snapshot.

Decision 2 — Warehouse stock: append-only ledger vs recomputed projection

Append-only stock_movements ledger
Chosen

Every receipt, dispatch, adjustment, and cancellation reversal is a signed row; balance = SUM, enforced non-negative by trigger.

  • Full audit trail (user + timestamp per movement) — fixes a named Excel defect
  • DB-level stock ≥ 0 guarantee (slide 24 non-negotiable)
  • Cancellation becomes an explicit reversal, fixing the current cancel bug
  • Slightly more code than a computed view; needs a balances view/index for reads
Recompute from orders + dispatches (Excel model)
Alternative

Keep stock as the pure projection ordered + received − dispatched, as Refresh Stock And Ordered does.

  • Simplest port; provably matches today's numbers
  • Cannot express manual adjustments, losses, or driver returns
  • No audit trail; no way to enforce non-negativity at write time

Decision: ledger. The projection formula survives as the on-order column (pending ordered comes from PO lines, not movements); physical stock comes from the ledger. This cleanly separates "on order" from "in warehouse" — a confusion the briefing explicitly calls out (slide 9).

Decision 3 — Replacing Power Automate + Excel templates

Server-side document generation + SMTP/API email
Chosen

Backend renders PO and dispatch documents (PDF via HTML template; XLSX via openpyxl where Excel output is still wanted), stores them in Azure Blob, and emails them directly.

  • Removes the 4 PA flows, their SAS-signed URLs, and the two "From PA" scripts entirely
  • Documents are archived and re-downloadable (today they exist only in sent mail)
  • Testable: golden-file document tests
  • Layout of the three templates must be reproduced once in code
Keep Power Automate as the delivery layer
Alternative

Backend POSTs to the existing PA flows, which keep filling the template workbooks.

  • Zero template work; familiar to ops
  • Keeps the most fragile part of the current system (signed URLs, 500-row batch deletes, workbook-open race conditions)
  • Two systems to operate forever

Decision: full replacement. The restock-verification "report" stops being a file at all — it becomes a page plus an email digest, since its only consumer is internal ops. The PO and dispatch sheets remain real documents because suppliers and drivers consume them externally. Whether drivers get PDF, XLSX, or both is an open question (Q3).

Decision 4 — Backend shape

Modular monolith (single FastAPI app)
Chosen

One deployable with routers/services per module and an in-process scheduler.

  • One box on Railway, one DB, trivial transactions across modules (dispatch confirm touches stock + documents + email)
  • Matches team size and 200-fridge scale for years
  • Scheduler shares the web process (acceptable at this load; extractable later)
Services split (API + worker + per-module services)
Alternative

Separate sync worker and API deployments, or full microservices.

  • Isolation of sync failures from API latency
  • Operational overhead unjustified at 40–200 fridges and a handful of users
  • Cross-service transactions complicate the dispatch-confirm flow

Decision: modular monolith. Reconsider only if Husky backfill/sync jobs measurably degrade API latency — the escape hatch is deploying the same image twice (web vs worker role flag), which the code structure keeps cheap.

Decision 5 — Forecast engine scope at cutover

Port the current algorithm (R1–R3) first and validate parity, then layer the briefing's enhancements (6–12-month windows, dual 50/50 scoring, residual-stock deduction via live DLC, weather adjustment) behind settings flags. Shipping the enhanced model untested against reality would make cutover disputes unresolvable — with parity first, every improvement is measurable against a trusted baseline. The final ambition level for v1 is an open question (Q4).

Manual verification

  1. After Phase 1 backfill: pick 3 random days; compare per-fridge sales counts in the app vs the Husky-driven numbers in Forecast V2's Added/Sold block — expect exact match.
  2. After Phase 2: create a real PO in staging with the recipient override; verify the emailed PDF is accepted by ops as equivalent to POTemplate.
  3. After Phase 3: run one delivery day fully in both systems; drivers use the new sheets; warehouse compares stock balances at end of day — expect zero unexplained variance.
  4. Edge cases: confirm a dispatch for yesterday (must demand explicit confirmation); receive more than ordered (must warn); cancel a received PO (stock must return); dispatch exceeding available stock (must block with cell-level errors, not save partially).
  5. After Phase 4: finance reproduces one month-end close in both systems and signs off the comparison.

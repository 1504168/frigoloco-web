Summary

Port the entire Frigoloco planning operation — currently 4 Excel workbooks, 26 Office Scripts, and 6 Power Automate flows orbiting "Smart Fridge Forecasting Tool V5.xlsx" — into a proper three-tier web application: a PostgreSQL database as the single source of truth, a FastAPI backend that owns the forecasting/scoring/dispatch/ordering engines and syncs the Intelligent Fridges vendor API, and a React single-page app whose 11 screens replace every sheet and button in the workbook. The front-end design is the primary deliverable of this spec: every screen is specified with layout, components, interactions, and the API calls behind it.

ScopeFull system port · greenfield app
StackReact + FastAPI + PostgreSQL
Screens11 screens · ~34 DB tables
Effort6 phases (~8–10 weeks)
RiskMedium-High — mitigated by parallel run

Why

The forecasting tool works, but it has outgrown Excel. The whole company's weekly cycle — forecast demand, compose menus, dispatch fridges, order from suppliers, verify restocks, report margins — runs through one workbook that only one person can safely touch at a time. Concrete pain points found during research:

  • Fragility by design. All 26 scripts address data by hardcoded cell coordinates and column indexes (e.g. row[15] = "Is Dispatched?", row[16] = "Week Start Date" in GlobalDispatchHistoryTable). Inserting one column silently breaks five scripts at once.
  • Security exposure. The vendor-API credentials (mathieu@frigoloco.be / password) are hardcoded in plaintext inside every API-calling script, and all 6 Power Automate flows are invoked via long-lived SAS-signed URLs embedded in script source.
  • Known latent bugs. Two different week-numbering conventions coexist (ISO-8601 "Jan-4" weeks in the dispatch roll-up vs "first Monday of the year" in the weekly ETL), which can misalign dispatch cost against sales revenue for the same week. A probable off-by-one exists between the weekly-summary writer (writes Catering Food Cost at column 13) and reader (reads column 12).
  • No concurrency, no audit. Sheets like Global Dispatch History (20,692 rows) are deleted-and-rewritten wholesale on every save; the only safety net is a manual full-sheet backup copy. There is no record of who changed what.
  • Duplication. The updateStockAndOrdered routine is copy-pasted verbatim into three scripts; category lists are re-declared per script and have already drifted (one script includes "10. Frozen Warm Dishes", its sibling does not).
  • Scale ceiling. Dispatch View is a 10,010-row × 93-column grid; Product Rating rebuilds from 365 days of API data on every run. Excel recalculation and Office-Script row limits (batched 500-row deletes because "PA cannot delete more than 1000 rows") are already visible in the code.

Cost of inaction: every new fridge, product, or workflow tweak increases the blast radius of a coordinate-indexed script edit, and the business's operational memory stays locked in one xlsx file on one OneDrive.

Known defects to fix during the port (not replicate)
  • Week numbering: standardize on ISO-8601 (see open question Q6). Today the dispatch roll-up uses ISO weeks while the weekly ETL anchors on "first Monday of the year".
  • Probable off-by-one between weekly-summary write (Catering Food Cost at index 13) and read-back (index 12).
  • Category list drift between the two "Pull Global Dispatch" scripts.
  • Vendor credentials + Power Automate SAS URLs in source — must move to environment configuration, and the vendor password should be rotated at cutover.

Proposed Solution

Build a three-tier web application. The backend becomes the single owner of all business logic (every formula above moves into one tested Python module each); PostgreSQL replaces every Excel table; scheduled sync jobs replace the "pull from API on button click" pattern; a document/email service replaces the Power Automate flows; and a React SPA replaces every sheet-as-UI. The Intelligent Fridges platform remains the upstream system of record for catalog, fridges, and telemetry — the app syncs it into local tables so screens are fast and history is queryable.

Vendor sync strategy

  • Nightly incremental sync (APScheduler): purchases, restocks, reviews for the trailing 3 days (idempotent upsert by tag/natural key — late events get captured); producttype + facility upsert.
  • On-demand sync: "Sync now" button on Admin, and the Forecast/Verification engines trigger a targeted sync of their window before computing, so results never depend on stale data.
  • Historical backfill: one-off management command pulling 365+ days in monthly chunks (mirrors what Update Rating does today) — this also serves spec 0001's "historical backfill" goal.
  • Live stock is never stored as truth — proxied with a 60-second cache for the Targets and Dashboard screens.

Cron layer (scheduled jobs)

APScheduler (AsyncIOScheduler) runs inside the FastAPI process lifespan. Every job records a run row (window, counts, status, error) and is safe to re-trigger manually from Admin → Sync. Full job specifications are elaborated in reference-docs/backend-and-cron-layers.md.

Front-End Design (primary focus)

Design principles

  • Grid-first. The users live in Excel today. Every planning screen centers on a high-performance editable data grid with keyboard navigation, copy/paste from Excel, column pinning, and fill-down — not on forms. Grid library per open question Q2 (default: AG Grid Community).
  • One workflow per screen, status always visible. Excel hid state in named cells ("Is Dispatched?", progress messages). Every entity here carries an explicit status chip (Draft / Saved / Sent / Dispatched / Verified…) and every long action shows progress inline.
  • Destructive-action confirmation. The workbook guarded "Create Individual Dispatch" behind a manual boolean. The app uses typed confirmation dialogs for actions with external side effects (emails to suppliers/logistics) showing exactly what will be sent and to whom, plus a global Test mode toggle that reroutes all email to a test address (replicating OrderSheetIsTestRun).
  • No refresh buttons for derived data. Stock position, margins, and totals are computed server-side from source tables; the UI just re-queries (TanStack Query invalidation).

Trade-off Analysis

Decision 1 — Database & backend stack

PostgreSQL + FastAPI (SQLAlchemy 2 / Alembic / Pydantic v2 / APScheduler) is chosen without a full bake-off: the team's existing tooling and skills are built around Python, FastAPI scaffolding, and PostgreSQL analysis workflows, and spec 0001 already targets a PostgreSQL database with vendor-API backfill. Any JVM/Node alternative would add a second ecosystem for zero functional gain on an internal tool. Risk accepted: Python is slower per-request than Go/Node, which is irrelevant at this user count (<10 concurrent users).

Decision 2 — Front-end framework

React 18 + Vite + TypeScript (SPA)
Chosen

Plain SPA served as static files by the backend; no server rendering.

  • Largest ecosystem for data grids (AG Grid, TanStack) and shadcn/ui
  • No SSR complexity — internal tool behind login, SEO irrelevant
  • Single deployable: FastAPI serves dist/
  • Initial bundle larger than a server-rendered first paint (acceptable on office networks)
Next.js
Alternative

React with file-based routing and SSR/RSC.

  • Conventions out of the box
  • Needs a Node runtime next to FastAPI — second server to operate
  • SSR/RSC machinery buys nothing for an authed grid-heavy tool
Vue 3 + Vite
Alternative

Comparable SPA capability.

  • Gentler template syntax
  • Smaller grid/component ecosystem for this exact Excel-replacement use case

Decision: React SPA (confirm via open question Q1). Reconsider only if the team standardizes on another framework elsewhere.

Decision 3 — Data-grid component (the make-or-break UI choice)

Decision: start with AG Grid Community; upgrade path to Enterprise (same API) exists if planners demand richer Excel-style range operations after go-live. Handsontable is the strongest pure-Excel feel but costs license fees on day one. Open question Q2 confirms.

Decision 4 — Vendor data: sync-to-DB vs live proxy

Sync into local tables (ETL)
Chosen

Nightly + on-demand incremental sync into append-only event tables; engines read local SQL.

  • Forecast/scoring over 365 days in milliseconds, not 12 API round-trips
  • History survives vendor retention limits; joins with app data (dispatches, orders)
  • Screens stay usable when the vendor API is down
  • Sync code + staleness window to manage (mitigated: engines trigger targeted sync of their window first)
Proxy live API calls per request
Alternative

Backend forwards each screen's needs to the vendor API (what Excel does today).

  • No sync machinery, always current
  • Reproduces today's slowness (365-day pull per rating refresh)
  • No joinable history, no offline resilience, rate-limit exposure

Decision: sync-to-DB. Exception: /stock/current stays a live proxied call (60s cache) because stock truth must be real-time and is cheap to fetch.

Decision 5 — Documents & email: replace Power Automate now vs keep it temporarily

Backend-native from day 1 (WeasyPrint PDFs + SMTP/Graph)
Chosen

The backend renders PO/dispatch/verification documents from Jinja2→PDF and sends email itself.

  • Kills all 6 SAS-URL flows and the template-workbook round-trips in one move
  • Documents become reproducible artifacts stored with their entities
  • Test mode enforceable in one place
  • PDF templates must be rebuilt to match the familiar Excel layouts (planner review needed)
Keep PA flows during transition
Alternative

Backend POSTs the same JSON payloads to the existing flows.

  • Zero document-fidelity risk at first
  • Keeps the SAS-URL secret sprawl and xlsx templates alive; two systems to debug; delays the cleanup that motivated the port

Decision: backend-native (confirm via Q4 — the fallback is wiring the old PA URLs behind the same service interface, which the design permits).

Decision 6 — Historical data migration scope

Options: (a) backfill vendor telemetry 365+ days from the API and import app history (2,593 order rows, 20,692 dispatch rows) from the workbook; (b) vendor backfill only, start app entities fresh; (c) start entirely fresh. Recommendation: (a) — stock position and "copy previous week" need dispatch/order history to be meaningful on day 1, and the import is a bounded one-off script. Cost: mapping the Excel free-text statuses (" Received ") onto the new enums. Decided via open question Q5.
